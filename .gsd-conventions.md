# GSD Conventions for the SKIDS Protocol Library

How GSD's milestone / phase / task model maps onto SKIDS protocol
authoring work.

---

## The mapping

```
GSD MILESTONE       =  A complete protocol from NOT_STARTED to APPROVED
GSD PHASE           =  A defined unit of work within that protocol
GSD TASK            =  A specific, scope-locked deliverable within a phase
```

This means each protocol = one GSD milestone. Each milestone has
multiple phases (drafting, review, integration). Each phase has 2–3
tasks maximum.

---

## Standard milestone structure for a Tier 1 protocol

A Tier 1 protocol moves through these phases:

### Phase 1: Scoping and evidence gathering

**Tasks:**
- Identify governing guidelines and citations
- Identify named protocol owner (subspecialist for governance)
- Confirm tier classification (Tier 1 vs. Tier 2 boundary)

**Deliverable:** §1 Service definition + §2 Evidence base + STATUS.md
with `IN_PROGRESS`

### Phase 2: Diagnostic workflow

**Tasks:**
- Author §3 Presenting concerns
- Author §4 Diagnostic workflow with structured history, exam, scales,
  device tests, decision tree

**Deliverable:** §3 + §4 complete

### Phase 3: Severity grading and therapeutics

**Tasks:**
- Author §5 Severity grading [SACRED — slow down]
- Author §6 Therapeutic options with weight-based dosing tables and
  in-clinic dispensary stock list

**Deliverable:** §5 + §6 complete; severity → tier mapping locked

### Phase 4: Hard referral triggers and follow-up

**Tasks:**
- Author §7 Hard referral triggers [SACRED — slowest section]
- Author §8 Follow-up schedule

**Deliverable:** §7 + §8 complete; §7 includes named partner
destinations and handoff packets

### Phase 5: Companion software specification

**Tasks:**
- Author §9 Companion software workflow with screen-by-screen specs
- Validate §9 with SKIDS Engineering for implementability

**Deliverable:** §9 complete; engineering sign-off captured in STATUS.md

### Phase 6: Parent and manager handoff

**Tasks:**
- Author §10 Parent-facing communication with scripted explanations
- Author §11 Manager handoff with subscription tier match and objections

**Deliverable:** §10 + §11 complete

### Phase 7: Quality and audit + final QA

**Tasks:**
- Author §12 Quality and audit
- Re-read entire protocol top to bottom for voice and consistency
- Mark all `[NEEDS EXPANSION]` tags intentionally; resolve unintended ones
- Update YAML frontmatter status to `DRAFT_COMPLETE`

**Deliverable:** Full protocol at DRAFT_COMPLETE; ready for editorial pass

### Phase 8: Editorial review

**Tasks:**
- Senior pediatrician completes editorial review per
  `workflows/how-to-review-a-protocol.md`
- Author addresses any feedback

**Deliverable:** Protocol at APPROVED; CLAUDE.md status tracker updated

### Phase 9: Companion deployment

**Tasks:**
- SKIDS Engineering integrates §9 specification into Companion Doctor Module
- Tested with at least one simulated patient case

**Deliverable:** Protocol at DEPLOYED; live in production Companion

## Standard milestone structure for a Tier 2 protocol

Tier 2 protocols (Endocrine, Neurology, Rheumatology) follow the same
9-phase structure but with two additional Phase 1 tasks:

- Confirm credentialed subspecialist as both author and protocol owner
- Confirm SKIDS Fellowship pathway language for §12

## How to start a new milestone

When GSD asks "what's the next milestone," reference `CLAUDE.md` §5
NEXT WORK ITEM. The milestone name should be the protocol_id:

```
Milestone: B1-05-skin
Description: Author SKIDS Skin Clinic protocol from NOT_STARTED to APPROVED
```

GSD then prompts for the first phase. Use the Phase 1 description above.

---

## How phases interact with STATUS.md

GSD's phase state lives in `.planning/`. STATUS.md is the human-readable
narrative state for the protocol.

The relationship:
- `.planning/` = "what GSD thinks is going on"
- STATUS.md = "what a human reader needs to know to resume"

Both are kept current. If they disagree, STATUS.md governs because
it's what humans review.

---

## Task scope discipline

GSD's `/gsd-plan-phase` typically generates 2–3 tasks per phase. This is
intentional — it's the maximum a single context window can produce
reliably.

If GSD proposes more than 3 tasks for a phase:
- Re-plan with smaller scope
- Or split the phase into two phases

If GSD proposes a single huge task:
- Break it into 2 smaller tasks
- Or accept it but plan to break mid-task at a clean boundary

This discipline is what prevents the "premature success" failure mode.

## Using `/gsd-progress` between sessions

Always run `/gsd-progress` first when opening a session. Output looks
like:

```
Current milestone: B1-05-skin
  Status: in progress (Phase 4 of 9)
Current phase: Hard referral triggers and follow-up
  Status: in progress (Task 1 of 2 complete)
Last completed: §7 Hard referral triggers — SACRED section reviewed
                with protocol owner Dr. X on 2026-MM-DD
Next task: Author §8 Follow-up schedule
Recommended action: /gsd-execute-phase
```

This is the orientation that takes the place of re-reading transcripts.

---

## Using `/gsd-extract-learnings` after a milestone

When a protocol reaches APPROVED, run `/gsd-extract-learnings`. This
captures:
- Patterns that worked well in this protocol's authoring
- Common pitfalls encountered
- Voice or structural patterns worth applying to the next protocol

The captured learnings become project-local skills that help converge
the library's voice and structure across protocols.

---

## When GSD and the SKIDS workflows disagree

If GSD's recommended next action conflicts with what
`workflows/how-to-write-a-protocol.md` recommends, the SKIDS workflows
govern. GSD is a workflow harness; the SKIDS workflows are the editorial
rules.

Common case: GSD may suggest moving forward when STATUS.md flags an
unresolved clinical question. In that case, resolve the question first
via STATUS.md communication channel, then resume GSD execution.
